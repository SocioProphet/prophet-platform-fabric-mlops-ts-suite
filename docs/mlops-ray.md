# Prophet Model Ops with Ray (Train + Serve)

> Ray is the default distributed runtime, but Prophet supports a broader MLOps ecosystem.
> See `docs/mlops-ecosystem.md` for the full “engines + contracts” view.


This document aligns Ray Train and Ray Serve with the Prophet Platform layering (PHT).

## Why Ray in Prophet?

Ray gives us:
- a distributed runtime for training, ETL, and hyperparam search,
- a first-class serving system (Ray Serve),
- and a Kubernetes-native control surface via KubeRay CRDs.

On Kubernetes, Ray recommends using the **KubeRay operator** and its CRDs:
- RayCluster
- RayJob
- RayService

## Core K8s atoms

### KubeRay operator (cluster add-on)
- Installed once per cluster (namespace `kuberay-system`).
- Provides CRDs and controllers.

### prophet-ray-serve (RayService)
- A RayService CR encapsulates a Ray cluster + a Serve application.
- You manage deploy/upgrade/status with `kubectl` like any other K8s resource.
- In RayService spec:
  - `serveConfigV2` holds the Serve deployment config (generated by `serve build`)
  - `rayClusterConfig` holds the RayCluster spec for head/worker pods

### prophet-ray-train (RayJob)
- RayJob can create a transient Ray cluster, submit the training job, and optionally delete the cluster when done.
- This is ideal for repeatable training runs and avoids keeping expensive GPU clusters running forever.

## Standard workflow (what happens end-to-end)

### 0) Data becomes a version
1. `prophet-ingest-gateway` receives events (ticks, features, graph events).
2. Events are written to the Prophet durability plane (commit log + immutable lake).
3. `prophet-lake-writer` publishes immutable dataset segments + a manifest.
4. A dataset version ID is created (e.g., `dataset:bars:v2026-01-14T12:00Z`).

### 1) Training run (Ray Train)
1. Prophet control plane schedules a training run (API call, cron, or CI trigger).
2. It creates/updates a **RayJob** (prophet-ray-train) with:
   - dataset version URI (manifest)
   - training code image (baked deps)
   - output URI for checkpoints/artifacts
3. Ray Train runs distributed training (prefer Train V2) and writes checkpoints to object storage.
4. The job reports metrics (accuracy, loss curves, etc.) and persists logs.

### 2) Evaluation + gating
1. A second RayJob (or a phase in the same job) runs evaluation on holdout data.
2. Prophet compares metrics against the currently promoted model.
3. If it passes gates, Prophet promotes the model to “candidate” or “production”.

### 3) Serving (Ray Serve)
1. Prophet updates a **RayService** (prophet-ray-serve) with the new model version:
   - new container image tag or new runtime config / model artifact URI
2. RayService performs an in-place update (Serve config update) and can do zero-downtime Ray cluster upgrades.

### 4) Rollouts (safe by default)
We do canaries at the **mesh / gateway layer**:
- deploy v2 side-by-side with v1 (separate RayService or separate route)
- use Istio traffic weights for 1%, 10%, 50%, 100%
- enforce SLO guardrails and auto-rollback

This avoids relying on app-internal traffic splitting mechanisms.

## Operational standards

- All training artifacts (checkpoints, metrics, configs) go to object storage.
- Every training run has a stable run ID and provenance:
  - code version (git SHA)
  - dataset version ID
  - hyperparameters
  - environment (container digest)
- All model serving endpoints emit:
  - request rate / latency / error rate
  - model version labels
  - business-level metrics when safe (e.g., PnL attribution in offline pipelines)

## Edge vs Core usage

- Core clusters:
  - GPU RayJob training
  - large RayService clusters for serving
- Edge clusters:
  - smaller RayService clusters (CPU) for low-latency inference close to users/data
  - training typically disabled (unless special edge training use cases exist)
